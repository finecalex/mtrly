## [Unreleased]

### Fixed
- **Extension panel missing on `mtrly.xyz` articles; balance poll broken after SPA navigation.** Two bugs: (1) `/api/match` looked up the raw normalized URL, so `mtrly.xyz/a/10` returned `match: false` even though the DB row has `circlearc-59513674.slonix.dev/a/10`. Fixed by canonicalizing own-domain aliases (`mtrly.xyz`, `www.mtrly.xyz`) to `NEXT_PUBLIC_APP_URL` before the DB lookup. (2) `pageManagedTimer` and `pageManagedInitialBalance` in `content.js` were never cleared on SPA navigation, so navigating between articles in the same tab left the stale balance poll running with the wrong initial balance. Both reset on URL change now. Extension `manifest.json` also adds `mtrly.xyz` to `host_permissions`.
- **Auto-cashout retry storm after a failed gateway withdrawal.** When `GatewayClient.withdraw()` threw, the balance was refunded and the very next tick (5s) re-triggered the threshold check, causing 50+ cascading `BalanceTransaction` failures. Root cause: the singleton `GatewayClient` enters a bad nonce state after a failed call and every subsequent attempt from the same instance also fails. Fix: (1) add a 60-second in-process cooldown per creator in `autoWithdraw.ts` so rapid retries are suppressed, (2) call `resetGatewayClient()` on failure so the next attempt gets a fresh client with a clean nonce. See `circlefeedback.md §4.7`.
- **Extension still billing during YouTube ads (regression).** `isYouTubeAdPlaying()` only checked `.html5-video-player`'s own class list, missing cases where YouTube places `ad-showing` on a parent/sibling element. Added a broad `document.querySelector('.ad-showing')` check as the first guard so any in-tree occurrence of the class stops the tick, regardless of which element YouTube annotates.
- **Refund entries in auto-cashout panel had no link and missing reason.** The "refunded" badge rendered as a non-clickable `<span>` when `explorerUrl` was null (Gateway not configured or no tx hash). Badge is now always an `<a>` — links to the specific arcscan tx when available, falls back to `https://testnet.arcscan.app`. "Why refunded:" reason box now renders for every failed entry (was gated on `failureReason != null`); fallback text "Cashout failed; balance has been restored to your account." shown for older entries that have no stored reason.

### Added
- **Threshold-based auto-cashout to creator EOA.** New `Balance.autoWithdrawThresholdUsdc` (Decimal nullable). When set, every tick that pushes the creator's balance over the threshold fires `maybeAutoWithdrawForCreator` from `applyTick` as a fire-and-forget background job. The job atomically zeroes the creator's balance + writes a pending `BalanceTransaction(type=withdraw, referenceId="auto-withdraw:pending")`, calls `Gateway.withdraw(amount, recipient=creator.ownedEoaAddress)`, then stamps the row with `auto-withdraw:onchain:<mintTxHash>` on success. On failure: refunds the balance and marks the row `auto-withdraw:failed-refunded`. Concurrent-tick safe (second tick sees zeroed balance and skips). One Gateway mint per "fill" — that's the per-creator batch the model needs.
- **Auto-cashout panel in creator dashboard.** New `<AutoCashoutPanel>` lives between the wallet-proof card and the publish form. Inline radio chips (off / $0.05 / $0.10 / $0.50 / $1.00) PUT to `/api/profile`. Below: "Recent auto-cashouts" list with timestamp, amount, and a green arcscan-linked tx-hash chip per cashout. `/api/creator/earnings` now returns `autoWithdrawThresholdUsdc` and `autoWithdrawals[]`.
- **Honest settlement footer on `/a/[id]`.** Replaces the misleading "Earnings settle to 0xCREATOR..." line (creator EOA usually shows zero on arcscan because batched settles route through the platform's Gateway pool until the creator withdraws). Now two lines: (a) "Each tap is a real USDC nanopayment on Arc Testnet, batched through Circle Gateway → view live settlement pool" linking to the platform Gateway address (100+ live transfers), (b) "Creator wallet (earnings withdraw here): 0x...".
- **mtrly.xyz custom domain wired in.** `docker-compose.yml` `VIRTUAL_HOST` and `LETSENCRYPT_HOST` lists now include `mtrly.xyz` alongside `circlearc-59513674.slonix.dev`. `www.mtrly.xyz` left out of the cert until its A record exists (Let's Encrypt fails the entire multi-domain request when one host has no DNS).

### Fixed
- **Extension panel debug logging.** Two `console.log` calls in `bootstrap()` so a creator inspecting devtools can see exactly what `/api/match` returned and whether the panel injection ran. Helps diagnose "the panel doesn't show on /a/N" reports without needing a remote console.

### Added
- **Per-article engagement analytics in the creator dashboard.** New `<EngagementOverview>` card mounts at the top of `/dashboard`. For each Mtrly article the creator owns, shows three stat tiles (avg engagement %, dropoff paragraph + dropoff %, total views) plus an SVG line chart of engagement % per paragraph index with a dropdown to switch articles. Backend: new `GET /api/creator/content-engagement?contentId=X` walks `Consumption.unitsConsumed` for every viewer that isn't the creator and computes the per-paragraph "share of viewers who reached at least paragraph i" curve. The dropoff marker highlights the steepest single-step drop. A second tab "Video" shows a "coming soon" panel — per-second engagement on YouTube/external requires per-second consumption tracking we don't have yet.
- **Extension panel auto-refreshes balance on Mtrly own pages.** When `/api/match` returns `pageManaged: true`, the extension's content script now starts a 3-sec poll of `/api/auth/me` that updates the panel's Balance and computes "this view" as `initialBalance - currentBalance`. So readers tapping paragraphs on `/a/[id]` see the meter move in their wallet panel without the extension needing to fire its own ticks.

### Fixed
- **Balance not decreasing on paragraph unlock when localStorage carries another viewer's set.** `<MeteredArticle>`'s paid-paragraph cache key was `mtrly:paid-paragraphs:<contentId>` — globally scoped, not per-viewer. After demo viewer A read several paragraphs, demo viewer B (same browser) saw those paragraphs as already-unlocked from the start, so clicking them was a no-op and `/api/billing/tick` never fired. Backend was fine; the meter was just silently skipping the network call. Key is now `mtrly:paid-paragraphs:<viewerId>:<contentId>` (or `:anon:` for guests). `<MeteredArticle>` now takes `viewerId: number | null` and the article page passes it through.
- **Browser extension overlay disappeared from Mtrly's own /a/[id] pages.** The previous fix made `/api/match` short-circuit to `match: false` for `kind === "mtrly"` — which stopped the extension from double-charging, but also stopped it from injecting the balance panel that gave readers continuous wallet context. Match now returns the actual content kind plus a `pageManaged: true` flag for Mtrly own URLs; extension content.js gets a third branch (`kind === "mtrly"`) that injects the panel + hydrates balance but does NOT start `setupTextPaywall`. Page-driven metering stays the source of truth.

- **Tap-to-reveal double-fire on Mtrly own articles.** `/api/match` was returning `kind: "web"` for our own `/a/[id]` URLs (because the kind is derived from the URL shape via `normalizeUrl`, not the DB row), which made the browser extension treat them as third-party web articles and run its `IntersectionObserver` paragraph paywall on top of the page's own click-to-reveal meter. Result: paragraphs unlocked themselves on scroll, and any paragraph the user actually tapped charged twice. `/api/match` now short-circuits to `match: false` whenever the resolved `ContentUrl.kind === "mtrly"`, leaving the page in full control of metering on its own surface.
- **Locked paragraphs containing inline links were unclickable.** `<MeteredArticle>` wrapped each locked paragraph in `<button>` with `dangerouslySetInnerHTML` for the body text — paragraphs that contained markdown links rendered an `<a>` *inside* the `<button>`, which is invalid HTML and let the anchor steal the click on Chrome/Safari. Replaced the `<button>` wrapper with a `<div role="button" tabIndex={0}>` and added `pointer-events-none` on the inner `<p>` so the wrapper is the only valid click target. Keyboard activation (Enter / Space) preserved.
- **Browser extension billed users while YouTube ads were playing.** YouTube uses the same `<video>` element for ads and content, swapping the source — so our 5-sec tick fired during ads too. Added `isYouTubeAdPlaying()` in `extension/content.js` that checks for the `ad-showing` / `ad-interrupting` classes on `.html5-video-player` and the `ytp-ad-*` overlay elements; `tickOnce` short-circuits when an ad is detected and resumes when it ends.

### Added
- **Per-paragraph paid-set persistence in `<MeteredArticle>`.** Server still only knows the count (`Consumption.unitsConsumed` is a counter, not an index set), so we now persist the actual paid-paragraph indices to `localStorage` under `mtrly:paid-paragraphs:<contentId>` and merge them with the server count on mount. Effect: if a user paid for paragraphs 1, 3, 5 and refreshes, those three stay open instead of "first 3 sequential" being unlocked. Sequential top-up still kicks in if localStorage is shorter than the server count (fresh device).
- **Admin `GET / PATCH /api/admin/content`.** Lets ops inspect every content row (including `rawUrl` which the public `/api/explore` hides) and rewrite a single row's URL / title / description / preview without touching FK-linked Payment, ViewSession, or Consumption rows. Used for one-off content fixes that have to preserve historical stats.
- **Admin `POST /api/admin/seed-creators`.** Takes a JSON list of demo creators (`{ name, slug?, bio?, avatarUrl?, urls[] }`) and provisions each: full user row + Circle Wallet + local EOA, optional onchain `depositFor` seed (defaults to $0.10), and registers each YouTube/web URL as a ContentUrl under that creator. Skips creators by existing slug and skips URLs already registered, so re-running is idempotent. Names are caller-supplied — endpoint deliberately doesn't ship a hard-coded list of "famous creator" identities.

- **Round 6: self-healing onchain settlement + extension popup activity.**
  - **Background onchain poller (`web/lib/onchainPoller.ts` + `web/instrumentation.ts`).** Next.js instrumentation hook starts a server-side loop that POSTs `/api/gateway/resolve` every 5 sec. The loop walks every Payment with a `nanopaymentTxId` but no `onchainTxHash` and (a) tries to populate the real Arc tx hash from Circle's `searchTransfers({status:"completed"})` map, (b) falls back to flipping `settledOnchain=true` so the UI stops reporting "batching…" for ticks Circle's v3 SDK can't surface a hash for. Self-healing — fresh ticks become explorer-linkable within ~5 sec of settlement without anyone clicking a button. Implementation goes via HTTP rather than direct SDK import to keep the Circle x402 batching SDK (which pulls Node `crypto`) out of the instrumentation webpack bundle.

Backfill walks 5 pages of completed transfers for historical reach. Pure idempotent: never overwrites an existing hash.

Existing dashboard recent-payments list automatically renders "onchain" for both.

### Added
- **Native articles — creators can write content directly on Mtrly.** Until now creators had to host content elsewhere (YouTube, their own blog) and register the URL. Now they can publish article-style text directly inside Mtrly, metered by the same per-paragraph extension flow.
  - `ContentKind.mtrly` enum + `ContentUrl.bodyMarkdown` (Postgres `Text`) — native articles are first-class ContentUrl rows. After insert, `rawUrl` and `normalizedUrl` are set to the canonical `https://<host>/a/<id>` so the extension's `/api/match` keeps working uniformly across kinds.
  - `lib/articleBody.ts` — `splitParagraphs()` (split on blank lines), `renderInline()` (XSS-safe escape + lightweight markdown: `**bold**`, `*italic*`, `[text](https://url)` external-only links), `firstParagraphPreview()`, `paragraphCount()`.
  - `POST /api/creator/content` — schema is now a Zod discriminated union `mode: "url" | "article"`. Article-mode validates body 20-50000 chars + creates the row + assigns canonical URL in a 2-step transaction. Returns `{articleUrl: "/a/<id>"}` for the dashboard's success card. Existing url-mode behavior unchanged for backward compat.
  - `PATCH /api/creator/content` — accepts `body` updates (only persisted if `kind == "mtrly"`).
  - `web/app/a/[id]/page.tsx` — public reader page (server component). Hero with hash-gradient + sparkle + title + description, creator strip with avatar + slug link, paragraph count + lifetime cost (`N paragraphs · ~$0.025 to fully read`), onchain-settlements badge. The body renders inside `<article>` with `<p data-mtrly-paragraph={i}>` so the existing extension's `<article> <p>` paywall logic kicks in unchanged.
